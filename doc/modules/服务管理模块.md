# 服务管理模块

> 返回 [项目架构深度分析](../项目架构深度分析.md)

---

## 一、模块定位

`service/` 包负责 **OpenCode serve 进程**的完整生命周期管理，包括进程启停、HTTP API 代理、SSE 事件流转发、项目树构建和终端启动。这是连接前端与 OpenCode 运行时的核心桥梁。

---

| 文件 | 职责 |

| `service/opencode/process.go` | 进程管理（启动/停止/健康检查/终端启动） |

## 三、进程管理（process.go）

### 3.1 核心结构

```go
type webSession struct {
    cmd      *exec.Cmd   // opencode serve 子进程
    port     int         // 服务端口
    hostname string      // 服务地址
}

var (
    WebSess     *webSession   // 全局单例
    WebSessMu   sync.Mutex    // 线程安全锁
    LastCfgHost = "127.0.0.1" // 最后已知配置
    LastCfgPort = 4096
)
```

### 3.2 启动流程

1. 参数校验（默认值 hostname=127.0.0.1, port=4096）
2. 检查已有进程：WebSess 非 nil 则返回当前状态
3. 检查外部运行的服务：`isOpenCodeServerRunning(hostname, port)`
4. 检查端口占用：`isPortInUse(hostname, port)`
5. 构建命令：`opencode serve --port X --hostname Y`
6. 注入代理环境变量：HTTP_PROXY / HTTPS_PROXY / ALL_PROXY / NO_PROXY
7. `cmd.Start()` → 轮询端口（250ms × 40 次 = 10 秒超时）
8. 超时则 `killProcTree`
9. 注册进程退出清理 goroutine
10. 返回健康状态 + 版本号

### 3.3 停止流程

1. 先停止 SSE 事件流
2. `taskkill /F /T /PID` 杀主进程 + 子进程树（bun 等）
3. 回退：`netstat` 查端口占用 → kill 对应 PID

### 3.4 健康检查

`getOpenCodeHealth(hostname, port)` 请求 `/global/health`：
- 2 秒超时 HTTP GET
- 解析响应中的 version 字段
- 返回 (状态, 版本, 是否在线)

### 3.5 终端启动

`LaunchWindowsTerminal(mode, webURL, dir)`：
- 优先查找 `wt.exe`（Windows Terminal）
- 回退到 `cmd /c start opencode`
- 支持 attach 模式（连接远程 OpenCode）和本地模式
- 可选指定工作目录

## 四、HTTP API 代理（service/opencode/api.go）

### 4.1 通用代理

`OpenCodeAPI(method, path, body)` 是通用 HTTP 代理，不限制端点：

```go
func OpenCodeAPI(method, path, body string) model.APIResult {
    base, err := getWebSessionBase()
    // 自动提取 body 中 "directory" 字段 → x-opencode-directory 请求头
    // 透传请求 → 返回状态码 + 响应体
}
```

前端通过 `api.OpenCodeCall(method, path, data)` 或直接 `api.OpenCodeAPI(method, path, body)` 调用。`OpenCodeCall` 自动完成 JSON.stringify/parse 和错误处理。`OpenCodeAPI` 会自动提取请求体中的 `directory` 字段并作为 `x-opencode-directory` 头传给 OpenCode，实现会话创建时的目录指定。

这意味着前端可以调用 OpenCode serve 的任何 API（/session, /agent, /message, /diff, /provider, /command 等），后端无需逐一封装。

### 4.2 项目树构建

`GetProjectTree(knownDirs)` 在 `service/opencode/api.go` 中实现：
1. 从 `/project` 获取项目列表
2. 从 `/session?roots=true&limit=500` 获取会话
3. 对每个已知目录查询 `/session?directory=X`
4. `buildTreeJSON()` 按 projectID → directory 两级分组返回 JSON

### 4.3 Question 工具应答

`AnswerQuestion(sessionID, answerLabel)` / `RejectQuestion(sessionID)`：
1. 获取 `/question` 列表
2. 按 sessionID 匹配问题
3. 调用 `/question/{id}/reply` 或 `/question/{id}/reject`

## 五、SSE 事件流（service/opencode/sse.go）

### 5.1 双通道架构

- **Wails 桌面端**：Go 逐行读取 OpenCode SSE → `wruntime.EventsEmit("oc-event", data)`
- **浏览器 Web 端**：Go SSE 订阅者广播 → HTTP `/events` SSE 端点 → 前端 `EventSource`

前端 `chat/events.js` 通过 `startEventStream()` 自动选择通道并监听处理。

### 5.2 实现细节

- 缓冲区：64KB 初始，最大 1MB
- 取消控制：`context.WithCancel` 支持优雅断开
- 重连：前端在 `oc-event-error` 时触发 `checkWebStatus()`

Go 端只负责把 OpenCode 的 SSE 数据透传到前端，不在服务层直接决定项目树刷新策略。项目树是否重建由前端 `chat/events.js` 根据事件类型和业务动作控制。

## 七、自动检测机制

`getWebSession()` 实现了透明自动检测：如果 WebSess 为空，自动用 LastCfgHost:LastCfgPort 检测外部运行的 serve，检测成功则自动创建 webSession 用于 API 调用。用户手动启动 `opencode serve --port 4096` 后，前端无需通过应用内按钮启动。

---

## 八、线程安全

所有涉及 WebSess 的操作通过 `WebSessMu sync.Mutex` 保护。`StartOpenCodeWeb` 采用先 Lock 检查、再 Unlock 执行、再 Lock 赋值的两段锁模式，避免长时间持锁。
