# OpenCode 管理中心 — 技术方案

## 项目定位

Wails v2 桌面应用（Go 后端 + 原生 JS 前端），为 OpenCode AI 编程助手提供可视化管理中心。当前系统已从单纯桌面应用扩展为：

- 桌面端 Wails 运行模式
- 浏览器可访问的页面 Web 服务
- Web 端目录浏览器与技能文件浏览/编辑能力
- 手机端工作区专用布局与长会话窗口化策略

## 三、架构分层

```text
前端（WebView2 / 浏览器）
  ├─ 原生 JS / CSS 变量主题 / 目录浏览器 / 技能文件浏览器
  ├─ 手机端抽屉布局 / 长会话窗口化渲染
  ├─ SSE 事件驱动消息缓存与增量更新
  └─ 25 个 JS 文件按功能分层（core / chat / filebrowser / views），零构建工具

Wails Runtime Bridge / Web app-call RPC（统一入口）

Go 后端（App 门面）
  ├─ internal/       跨包共享工具（symlink / pathutil / executil / fileutil）
  ├─ config/         配置读写（provider / omo / skill / commands）
  ├─ service/        业务逻辑（opencode / filebrowser / web）
  ├─ model/          前后端共享类型
  └─ app_dispatcher  AppCall 统一分发（~400行，覆盖全部方法）
```

工作区采用 **SSE 事件驱动 + 前端消息缓存/增量渲染**。前端 JS 按功能分层存放于四个子目录：
- `core/` — 基础层：主题、工具、全局状态、API 代理
- `chat/` — 聊天工作区：12 个模块（navigation / commands / mobile / search / config / service / tree / sidepanel / cache / render / events / session）
- `filebrowser/` — 文件浏览器：preview / browser / dir
- `views/` — 独立视图：omo-config / skill-manager / commands / provider

Web 端通过统一 `/api/app-call` RPC 调用后端，不再有独立 REST 路由。

## 六、构建与执行约束

项目使用 Wails CLI 构建，`wails.json` 位于**项目根目录**。因此构建命令必须在仓库根目录执行，例如：

```powershell
wails build
```

如果在 `build/bin` 等子目录执行，会因为找不到 `wails.json` 而直接失败。这属于执行目录问题，不是源码编译错误。
